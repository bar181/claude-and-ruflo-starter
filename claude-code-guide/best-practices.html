The #1 Rule

Claude Code is an agentic coding environment — it reads files, runs commands, makes changes, and works through problems autonomously. But its context window fills up fast, and performance degrades as it fills. Every message, file read, and command output consumes context. Managing this single resource is the highest-leverage skill you can develop. The five best practices below all flow from this constraint.

✓ 1. Give Claude a Way to Verify Its Work

This is the single highest-leverage thing you can do. Claude performs dramatically better when it can check its own output — run tests, compare screenshots, validate results. Without verification, you become the only feedback loop.

Before (Vague)

  • “implement a function that validates email addresses”
  • “make the dashboard look better”
  • “the build is failing”

After (Verifiable)

  • “write validateEmail. Test: user@example.com → true, invalid → false. Run tests after.”
  • “[paste screenshot] match this design. Screenshot result & compare.”
  • “build fails with [error]. Fix root cause & verify build succeeds.”
Verification methods: test suites, linters, type checkers, bash commands, screenshots (Chrome extension), expected input/output pairs. The better Claude can self-check, the less you intervene.

📋 2. Explore First, Then Plan, Then Code

Letting Claude jump straight to coding often produces code that solves the wrong problem. Separate exploration from execution using Plan Mode.

1

Explore (Plan Mode)

Enter Plan Mode (Shift+Tab twice). Claude reads files and answers questions without making changes. Ask it to understand the existing code.

2

Plan (Plan Mode)

Ask Claude to create a detailed implementation plan. Press Ctrl+G to open the plan in your editor for direct editing before Claude proceeds.

3

Implement (Normal Mode)

Switch back to Normal Mode and let Claude code, verifying against its plan. Include tests and validation in the same prompt.

4

Commit

Ask Claude to commit with a descriptive message and open a PR. It works directly with git.

When to skip planning: If you could describe the diff in one sentence (fixing a typo, adding a log line, renaming a variable), skip the plan. Planning is most useful when the approach is uncertain, multiple files change, or you’re unfamiliar with the code.

🎯 3. Provide Specific Context in Prompts

The more precise your instructions, the fewer corrections you’ll need. Reference specific files, mention constraints, and point to existing patterns.

StrategyExample
Scope Specify file, scenario, testing prefs “write a test for foo.py covering the edge case where user is logged out. avoid mocks.”
Source Direct Claude to the right source “look through ExecutionFactory’s git history and summarize how its API evolved”
Pattern Point to existing code patterns “look at HotDogWidget.php. Follow that pattern for a new calendar widget.”
Symptom Provide symptom + likely location “login fails after session timeout. check src/auth/ token refresh. write a failing test, then fix.”
Rich content methods: @file.js to include files · paste/drag images · provide URLs · pipe data with cat error.log | claude · tell Claude to fetch what it needs with Bash, MCP, or file reads.

📈 4. Manage Context Aggressively

Context is your most important resource. When it fills, Claude starts “forgetting” earlier instructions or making more mistakes.

ActionWhat It Does
/clearReset context entirely between unrelated tasks — the simplest and most effective action
EscStop Claude mid-action, context preserved, redirect immediately
Esc + EscRewind to a previous checkpoint (conversation, code, or both)
/compact <focus>Manually compress context with optional focus instructions
/btwAsk a quick question in a dismissible overlay — never enters conversation history
SubagentsDelegate research to separate context windows that report back summaries

⚙ 5. Configure Your Environment

A few setup steps make Claude Code significantly more effective across all sessions.

A

Write an Effective CLAUDE.md

Run /init to generate a starter file. Include only things Claude can’t infer from code: build commands, code style deviations from defaults, testing preferences, repo etiquette, and common gotchas. For each line, ask: “Would removing this cause Claude to make mistakes?” If not, cut it.

B

Set Up Permissions

Use Auto Mode (classifier reviews commands), allowlists (permit known-safe tools), or sandboxing (OS-level isolation) to reduce interruptions while staying safe.

C

Install CLI Tools

Install gh for GitHub integration (issues, PRs, comments). Claude can also learn unfamiliar CLIs on the fly: “Use foo --help to learn about it, then use it to solve X.”

D

Connect MCP Servers

Run claude mcp add to connect Notion, Figma, databases, Slack, and other external tools.

E

Create Skills & Hooks

Skills (.claude/skills/) package reusable workflows. Hooks run deterministic scripts at specific points — unlike CLAUDE.md instructions which are advisory, hooks guarantee the action happens.

CLAUDE.md tip: If Claude keeps ignoring a rule, the file is probably too long and the rule is getting lost. If Claude asks questions answered in CLAUDE.md, the phrasing might be ambiguous. Treat it like code: review when things go wrong, prune regularly.

🚀 Scaling: Parallel Sessions & Automation

Once you’re effective with one Claude, multiply your output.

  • Desktop App Manage multiple local sessions visually, each in its own isolated worktree
  • Web Sessions Run on Anthropic cloud infrastructure in isolated VMs — no local setup
  • Agent Teams Automated coordination of multiple sessions with shared tasks and messaging
  • Fan-Out Scripts Loop through files calling claude -p for each, with --allowedTools to scope permissions
  • Writer/Reviewer Session A implements, Session B reviews with fresh context (no bias toward its own code)
  • Scheduled Tasks Cloud or desktop schedules for PR reviews, CI failure analysis, dependency audits
# Fan-out example: migrate 2,000 files in parallel
for file in $(cat files.txt); do
  claude -p "Migrate $file from React to Vue. Return OK or FAIL." \
    --allowedTools "Edit,Bash(git commit *)"
done
